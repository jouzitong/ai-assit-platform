== 规范说明

本规则用于约束 `STANDARD_DEV_FLOW` 的执行门禁与裁决标准。

Flow 权威定义：

- `.codex/DSL/flows/standard-dev-flow.adoc`

== 总体约束

- 必须按步骤顺序执行，不得跳步。
- 每一步必须有可追溯产物。
- 每一步失败必须明确失败原因与修复动作。
- 文档检查是强门禁，不是可选项。

== 分步门禁规则

=== Step 1：INTAKE（口语化需求澄清）

必填产物：

- 背景与目标
- 范围与排除项
- 依赖与约束
- 歧义清单

通过条件：

- 关键歧义项已裁决或明确责任人/时间点

失败条件：

- 目标不明确且无法裁决
- 核心约束缺失

=== Step 2：REQUIREMENT_SPEC（标准需求文档）

必填产物：

- 标准需求文档
- 验收标准（可量化/可验证）

通过条件：

- 验收标准可直接用于测试裁决

失败条件：

- 验收标准不可验证或相互冲突

=== Step 3：PROTOTYPE_DESIGN（原型设计）

必填产物：

- 流程说明
- API 草案
- 数据状态变化说明
- 异常场景说明

通过条件：

- 原型与需求一致，且可支持任务拆分

失败条件：

- 原型无法支撑实现或存在重大缺口

=== Step 4：PLANNING_OWNERSHIP（任务规划与责任划分）

必填产物：

- Task/Sub Task 列表
- depends_on 依赖关系
- owner 责任划分

通过条件：

- 每个 Sub Task 有明确 owner
- 依赖关系可裁决

失败条件：

- owner 缺失
- 依赖关系不完整或循环依赖

=== Step 5：IMPLEMENTATION（系统开发）

必填产物：

- 代码变更记录
- 执行日志（ANALYZE/PLAN/EXECUTE/VALIDATE/REPORT）
- 文档同步更新记录

通过条件：

- 实际改动与计划一致或偏差可解释
- 文档同步完成

失败条件：

- 关键改动无验证证据
- 文档未更新或更新与代码不一致

=== Step 6：SYSTEM_TEST（系统测试）

必填产物：

- 测试范围
- 测试结果与证据
- 失败根因与修复记录（如有）

通过条件：

- 关键路径通过
- 验收标准全部满足

失败条件：

- 关键用例失败
- 无法提供测试证据

=== Step 7：DELIVERY_DECISION（需求完成裁决）

必填产物：

- 交付结论（DONE/BLOCKED）
- 残余风险
- 后续行动项（如有）

通过条件（DONE）：

- 测试通过且证据完整
- 必要文档齐全且一致

失败条件（BLOCKED）：

- 关键验收项未通过
- 文档缺失或关键信息不一致

== 文档强制检查清单

涉及组件能力时，以下文档必须同步检查：

- 系统组件详解文档
- 快速开发文档
- 版本升级文档（若变更影响兼容性或升级路径）

== 冲突处理

若与其他规则冲突，优先级如下：

1. `.codex/DSL/rules/00-global-rules.adoc`
2. 本文件（`06-standard-dev-flow-rules.adoc`）
3. 其他流程摘要规则
