# 页面结构规范

## 1. 目标

- 统一前端页面目录结构，保证页面职责清晰、定位明确、易于复用。
- 一个页面模块必须把路由、页面视图、数据逻辑、样式组织在同一目录下，避免文件散落。
- 页面以 `menu-key` 为最小业务单元进行组织，便于菜单、路由和代码目录一一对应。
- 一个菜单目录下可以包含多个子页面，每个子页面都按 `service/`、`views/`、`styles/`、`data/` 四类目录拆分。

## 2. 目录结构

页面目录统一按以下结构组织：

```text
src/
  views/
    domain/
      {menu-key}/
        service/
        views/
        styles/
        data/
        index.vue
```

## 3. 文件职责

- `service/`：子页面专属的数据请求、接口编排、提交逻辑、异步流程封装。
- `views/`：子页面纯视图层，只负责模板渲染、事件触发和状态展示。
- `styles/`：子页面样式，只写当前页面的结构和局部样式，不写主题值。
- `data/`：子页面静态数据、默认配置、表格列、表单项、字典映射等纯配置。
- `index.vue`：页面入口和聚合层，负责组合 `service`、`data`、`views` 和样式。

## 4. 组织原则

- 页面目录必须和路由、菜单语义一致，`menu-key` 要稳定且可追踪。
- `index.vue` 是对外唯一入口，路由只指向它，不直接指向内部子文件。
- 视图、数据、样式、服务四类职责必须拆开，不允许把所有逻辑写进一个 `.vue` 文件里。
- 同一个菜单下的多个子页面分别落在对应的 `service/`、`views/`、`styles/`、`data/` 目录中，不要混放。
- 页面内共用的业务状态优先放在 `index.vue` 或子页面的 `service/` 中，不要散落在多个组件中。

## 5. 拆分边界

- `service/` 只处理数据流和业务流程，不写 DOM 结构。
- `views/` 只接收输入和触发事件，不直接耦合接口实现。
- `styles/` 只描述布局和样式，不定义主题常量。
- `data/` 只提供静态结构和默认值，不发起请求。

## 6. 使用要求

- 新增菜单前先建 `menu-key` 目录，再补路由和菜单映射。
- 新增子页面时，先在 `service/`、`views/`、`styles/`、`data/` 下补对应目录，再接入 `index.vue`。
- 页面复杂度上升时，优先继续向子组件拆分，而不是继续膨胀 `index.vue`。
- 如果一个 `menu-key` 目录下的子页面越来越多但没有清晰分组，说明页面边界已经不清晰，应重新拆分。
