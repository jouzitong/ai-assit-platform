# 组件规范

## 1. 基本原则

- 一个组件只承担一类职责，避免展示、交互、请求、状态编排混在一起。
- 公共组件尽量无业务耦合，输入输出明确，便于跨页面复用。
- 页面级复杂逻辑优先拆成多个小组件，再由页面组合。
- 页面模块目录结构和页面职责拆分，统一遵循 [页面结构规范](./page.md)。

## 2. 组件分层

- 页面组件放在 `views/`，负责页面级组装。
- 布局组件负责壳层、导航和区域结构，不直接承载业务细节。
- 通用功能组件负责表单、弹窗、列表、卡片等可复用能力。

## 3. 页面模块内部拆分

- 页面模块按 `service.js`、`views.vue`、`styles.scss`、`data.js`、`index.vue` 拆分，各司其职。
- `index.vue` 作为页面入口，负责把服务、数据、视图和样式组合起来。
- 如果页面只有很少逻辑，也要保留目录边界，不能把所有内容重新压回单文件组件。

## 4. 组件接口

- 组件 `props` 要明确、稳定，避免传入大而全的对象后在内部随意解构。
- 组件事件命名要表达动作语义，不要用含糊的通用事件名。
- 组件内部状态尽量最小化，能由父组件控制的状态不要重复维护。

## 5. 复用要求

- 可复用组件应先抽象共性，再接收必要的配置，而不是复制页面代码。
- 复用组件的默认行为要合理，外部传参只覆盖差异部分。
- 不要为了单次使用而提前过度抽象，先保证职责清晰，再逐步复用。

## 6. 交互与状态

- 组件要显式处理加载、空、成功、失败等状态。
- 弹窗、抽屉、表单等组件应可控、可关闭、可重置，避免状态残留。
- 长流程交互优先拆成步骤清晰的组件链路，不要堆进一个大组件。

## 7. 维护要求

- 新增组件前先判断是否已有可复用实现，避免重复造轮子。
- 删除组件时同步清理引用，避免路由、页面和样式残留。
- 复杂组件需要保留简短注释，说明关键交互和数据流。
