# 服务模块结构规范

## 1. 目标

- 统一后端服务的基础模块结构，降低新服务初始化和后续维护成本。
- 先约束通用骨架，再按实际业务补充领域模块，避免一开始过度拆分。
- 保持模块职责清晰，避免把启动、契约和业务实现混放。

## 2. 基础模块

一个后端服务至少包含以下基础模块：

- `boot`：启动层，只负责应用启动、配置装配、组件扫描和基础集成。
- `core`：核心业务实现层，承载控制器落地、应用服务、领域编排和业务逻辑。

如果该服务需要对其他服务提供内部调用接口，则新增：

- `api`：接口契约层，只放对外暴露的内部接口、DTO、枚举和契约定义，不放业务实现。

## 3. `api` 模块使用规则

- 只有在服务确实需要对外提供接口时，才新增 `api` 模块。
- `api` 模块中的接口统一视为内部接口，不直接作为公网开放接口约定。
- `api` 模块只定义契约；具体实现放在 `core` 模块中。
- `boot` 模块负责把 `api` 和 `core` 装配成可运行服务，但不承载业务逻辑。

## 4. 内部接口前缀规范

- 需要对外提供的内部接口，统一使用前缀 `(/internal/{version:v1})`。
- 默认版本从 `v1` 开始；后续出现不兼容调整时，再按实际情况演进版本号。
- 新增内部接口时，先复用已有版本前缀，不要为单个接口随意创建独立风格的路径。

## 5. 扩展模块原则

- 在 `boot`、`core` 之外，其余模块都属于按业务实际情况补充的扩展模块。
- 是否新增扩展模块，取决于当前服务是否已经形成清晰、稳定、值得独立维护的业务边界。
- 扩展模块命名应直接反映业务语义，避免引入模糊或一次性目录。
- 如果只是少量功能点，优先先落在现有 `core` 中，避免提前拆出没有边界的空模块。

## 6. 落地原则

- 先确定服务是否需要对外提供内部接口，再决定是否建立 `api` 模块。
- 先保持 `boot`、`core` 职责干净，再考虑扩展模块拆分。
- 新增模块时，目录、命名和依赖方向必须保持单向清晰，避免循环依赖和职责倒挂。
